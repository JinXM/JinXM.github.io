加密

引子

战争时期，我方需要用书信/电报进行传输信息，但又怕被中途被敌军截获，那怎么办呢？于是大家想到了一个办法，把文本内容通过一定规则变换后再传输，那样就算被对方截获，对方也读不懂。如果安全送到我方人员手里，因为事先沟通好了规则，所以是能读出信息的。

这就是一个简单的加密，而加密的目的是：保证信息传输的安全。

加密和解密

抽象为模型。

加密是 cyt = E(msg, k1)，msg是需要被加密的原文，k1是加密参数（密钥），E()是加密的算法，cyt 是加密的结果（密文）。

解密是 msg = D(cyt, k2)，D()是解密的算法，k2是解密的参数（密钥）。

对称加密

比如我方情报加密，使用的是给每个字母向后挪3位，尾字母向首字母挪。a->d、b->e…x->a、y->b、z->c。那加密算法E()就是字母向后挪k1位，3就是加密密钥k1。

但是如果敌方也知道了我们的加密算法和密钥，并截获了一份情报。那么他们就可以通过把情报内容向前挪3位获取到原文。这时候解密算法D()就是字母向前挪k2位，3就是解密密钥k2。

这种可以通过加密算法，可以由k1反向推算出解密密钥k2的加密方式叫做对称加密。

也即：加密解密使用同一个密钥

非对称加密

我军之前因为使用对称加密，被内奸透露了加密方法和k1，导致情报泄露伤亡惨重。

于是决定改进算法，即使对方知道了cyt = E(msg, k1)的算法和k1，也不能推出解密密钥k2。这样发信人使用E(msg, k1)，接信人使用只有自己知道的k2。有多个部队知道加密算法和k1，给司令写信，即使泄露中间被截取，敌方也不能获取到具体信息。因为解密的解密密钥k2只有司令一个人知道，并且无法通过k1推算出来。但是司令通过k2去加密的信息，却能被k1解密出来，保证了司令的命令能传达到各个部队。

这种通过加密密钥k1加密，只能通过解密密钥k2解密，由k2加密的内容，却能被k1解密，并且不能由 k1推算出k2的算法叫做非对称加密。

也即：加密解密使用2个不同的密钥，私钥加密的只能由公钥解密，公钥加密的只能由私钥解密。

推论

  • 对称加密要生效，k1=k2必须严格保密
  • 非对称加密要生效，写信方的k1(公钥)可以不保密，但是收信方的k2(私钥)必须保密
  • 对于非对称加密，A 持加密密钥（公钥）k1，B 持解密密钥（私钥）k2
    • A加密一段 msg给 B，B 能解开的话。则可推断出，A 一定是密钥k1的拥有者
    • B 用公钥加密一段msg2给 A，则其他人都看不到原文，能解密的一定是 A。

认证

另一个引子

有一群自发的军队要加入我军，但是他们之前没有接触过我军。在路途中，他们遇到了敌军并先说了自己要干嘛，于是敌军机智的假冒成我军，收编了了这只军队。

这里就引出了一个互联网通信的问题：身份伪造。

客户端通过DNS 解析，获取访问域名的ip 地址，与server 建立连接通信。但是如果DNS 解析结果被串改了到了伪造server，即使有了加密的传输，因为一开始建立连接的对象是假的，所以你的私密信息比如密码什么的，会是有很大风险被窃取。

因此上，就需要一个证明该身份是该身份的机制，这就是证书。

Certificate Authority

CA（Certificate Authority），数字证书认证机构，即颁发证书的组织。

介绍 Domain Validated (DV) SSL，域名认证证书 Organization Validated (OV) SSL，组织/企业认证证书 Extended Validation (EV) SSL，增强型认证证书

比如你大学毕业，学校给你发毕业证书。证书证明了：你读过大学，并顺利毕业。但是也有的人，通过找一些黑组织，买了假的毕业证书。

那么，证书证明了你的身份，又怎么证明这个证书是真的证书呢？

证书是一个树状结构，顶层是根 CA 认证中心，以后每一层的认证中心/证书，都是由上一层签发。

我们使用数学归纳法证明：

  1. 根 CA 认证中心是可信赖的。
  2. 根 CA 认证中心签发了第二层的证书。
  3. 因为第二层身份的证书是根用密钥签算出来。根据前文加密的推论中可知，用根的公钥可以解开第二层的证书，那么就可以证明第二层的证书是根签算的。
  4. 根给第二层证书，证明了第二层是可信赖、权威的。
  5. 同理，第二层给第三层发证书，证明了第三层是可信赖、权威的。
  6. 依次类推，第 N 层的证书是可信赖的。

摘要算法

比如我写了一封信，但是我怕送信的人修改了其中的内容，于是我打算在信的尾巴加一段验证文字，里边记录着这封信写了多少个字、用什么语言、多少个段落等信息，只要信的主要内容被修改过，验证文字的内容一定对不上。这样只要验证文字没有被修改过，就能通过验证文字来确定信的内容是否被修改过。

上边的验证文字就是一个简单的摘要，摘要的目前是校验数据（未被修改、未丢失、未增加）。

而摘要算法则是生成这段摘要的方法，通过主体数据生成一段固定长度的摘要。摘要算法本身并不具备加密功能，加密则是由证书签名实现。

证书签名

如何证明一个证书是可信的呢？即证明证书是它的颁发者颁发的。

一个证书里存了很多字段，这里说3个：公钥、证书内容、加密算法、证书内容的加密文(由上层证书私钥加密)。

证明的方法是：

还是利用之前加密的推论。

  1. 证明证书内容的加密文能被上层的公钥解开，解开后内容为msg1，表示msg1和公钥都是权威认证可靠的。
  2. msg1和证书里明文的证书内容相同，那么证明明文的证书内容是可信的。

推论

  • 只要根 CA 可信，往后每个由上层颁发过证书的机构/证书拥有者，其被证书证明的身份都可信

SSL/TLS

SSL/TLS 是加密传输协议。HTTPS 传输就是依赖于 SSL/TLS 协议完成信息安全的传输。



Phase 1

  1. Client:
    • generate random number: RNc
    • client_hello: 向 Server 发消息，告诉Server自己支持的SSL版本、加密算法等，和随机数 RNc
  2. Server:
    • 接收客户端client_hello 消息，并generate random number RNs
    • server_hello: 向Client 回应报文，包含服务端支持的 SSL 版本和加密算法等，和第二个随机数 RNs
  3. Client
    • 接收服务端server_hello 消息
  4. 此时Client 和 Server 都:
    • 都拥有2个随机数 RNc 和 RNs
    • 都知道彼此支持哪些SSL 版本和加密协议

Phase 2

  1. Server:
    • 向Client 发送certificate 消息，消息内容主要是一个包含 Server 公钥的证书
    • demand client certificate：发消息给client要求client 传证书证明自己（可选）
  2. Client:
    • check server certificate: 使用上文的证书验证方法，验证证书的正确性
  3. 此时:
    • Server 向 Client证明了自己的身份（没有被伪造）
    • Client 知道了 Server 的公钥

Phase 3

  1. Client:
    • client certificate: 给 Server 发证书，证书内容是一个包含 Client 公钥的证书（可选）
  2. Server:
    • check client certificate: 验证Client证书的正确性，确认Client 身份
  3. Client:
    • hash over all previous messages(signed with key): 使用自己私钥加密之前的通讯信息（为了证明Phase1、Phase2通话的是自己）
  4. Server:
    • check hash and signature: 校验数字签名，确认 RNc、RNs以及 Client 的公钥的有效性
  5. Client:
    • generate random number pre-master-secret：生成第三个随机数PMS
    • 向Server发送随机数 PMS(被 Server的公钥加密)
  6. 此时Client 和 Server:
    • 都有三把钥匙：自己的私钥、自己的公钥、对方的公钥
    • 都有3个随机数：RNc、RNs、PMS
    • 都使用3个随机数和约定好的算法(Phase 1中沟通过)生成本次通话的会话密钥MS

Phase 4
前3个步骤已经完成了 SSL 加密通讯的建立，Phase 4就是加密通讯的过程：

  • 都使用会话密钥MS进行加密通讯，属于对称加密

矛盾

之前介绍说，证书的作用是证明身份，而且一个可信赖的证书本身不能被伪造。

通讯中证书的作用是防止“身份伪造”。

因为 CDN 是对 Client 是透明了，Client 认为自己直接请求的是源站 Source。那么 配置了 HTTPS 的Client 会要求 Source 出事证书证明它是它（比如证书里有域名）。但是如果再配置了 CDN，中间的 Cache 服务器是没有这个证书的，就会被 Client 认为是伪造的源站，从而没办法正常访问。

解决

国内厂商一般的解决方法是：

让用户提供他们的证书和密钥，然后厂商再把这些证书分发到加速Cache节点。

这样一来，Cache也具有了能够提供证书的能力了。
